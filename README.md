## 1. Хамгаалалтын үндсэн ярианы логик

Хамгаалалтын үед слайд дээрх бүх текстийг унших хэрэггүй. Харин дараах логикоор ярь:

1. **Асуудал**  
   Үйлдвэрийн агуулах, үйлдвэрлэл, аюулгүй байдал, ложистик, тоног төхөөрөмжийн мэдээлэл олон газар тархсан тул удирдлага хурдан шийдвэр гаргахад хүндрэлтэй.

2. **Шийдэл**  
   Эдгээр мэдээллийг нэг KPI dashboard дээр нэгтгэж, захирал системийн нөхцөл байдлыг 5 секундэд ойлгох боломжтой болгосон.

3. **Техникийн хэрэгжилт**  
   Next.js App Router, TypeScript, Prisma ORM, MySQL/Aiven Cloud, JWT, RBAC, Zod validation, Vercel deployment ашигласан.

4. **Код ба хамгаалалт**  
   Нэвтрэлт, эрхийн шалгалт, protected API route, input validation, CRUD, dashboard refresh, audit log, OWASP эрсдэлийг харгалзсан.

5. **Үр дүн**  
   Төсөл нь зөвхөн UI mockup биш, өгөгдлийн сан, API, authentication, authorization, deployment, KPI visualization-тэй fullstack веб систем болсон.

**Онцлох өгүүлбэр:**

> Энэ слайд дээр гурван хувилбарыг харьцуулсан. Excel болон цаасан бүртгэл нь эхний шатны бүртгэлд тохиромжтой боловч real-time dashboard, role permission, audit log, API protection зэрэгт сул. Ерөнхий dashboard систем нь chart харуулж болох ч уул уурхайн тэсрэх бодисын үйлдвэрийн онцлог болох safety, material threshold, RPM telemetry, хэлтсийн role-based flow-г тусгайлан шийдэхгүй. Миний систем эдгээрийг domain-specific байдлаар нэгтгэсэн.

**Юуг тайлбарлах вэ:**

- Excel/цаасан бүртгэл: энгийн боловч manual, алдаа гарах магадлалтай.
- Ерөнхий dashboard: visual талтай боловч domain workflow сул.

**Онцлох өгүүлбэр:**

### Хуудас 6. Системийн архитектур

**Слайдын зорилго:**  
Системийн ерөнхий техникийн урсгалыг тайлбарлах.

**Юу гэж хэлэх вэ:**

> Архитектурын хувьд энэ төсөл modular monolith хэлбэртэй. Өөрөөр хэлбэл frontend, API route, business logic нэг Next.js codebase дотор байгаа. Browser-оос хэрэглэгч request илгээнэ, Next.js App Router API route хүлээн авна, дараа нь authentication, RBAC, validation шалгаад Prisma ORM-оор Aiven MySQL database-тэй ажиллана.

**Юуг тайлбарлах вэ:**

- `Browser`: хэрэглэгчийн dashboard, table, form.
- `Next.js App Router`: page болон API route.
- `JWT`: login session шалгалт.
- `RBAC`: ADMIN, MODERATOR, USER эрхийн шалгалт.
- `Zod`: input validation.
- `Prisma`: database query.
- `Aiven MySQL`: cloud database.
- `Vercel`: production hosting.

**Онцлох өгүүлбэр:**

> Modular monolith сонгосон шалтгаан нь дипломын төслийн хүрээнд нэг repo, нэг deploy pipeline, нэг TypeScript codebase илүү ойлгомжтой бөгөөд хурдан хөгжүүлэх боломжтой байсан.

**Асуувал богино хариулт:**

Асуулт: "Monolith сул талтай юу?"  
Хариулт: "Хэт том системд service тусгаарлах шаардлага гарч болно. Гэхдээ энэ төслийн scope-д modular monolith нь complexity бага, deploy энгийн, maintenance ойлгомжтой."

### Хуудас 7. Өгөгдлийн сан ба ERD

**Слайдын зорилго:**  
Database structure, entity relation, Prisma ORM-ийн үүргийг тайлбарлах.

**Юу гэж хэлэх вэ:**

> Энэ ERD дээр системийн гол өгөгдлийн холбоос харагдаж байна. Төв entity нь User бөгөөд User нь Department-тэй холбогдоно. Department нь агуулах, үйлдвэрлэл, аюулгүй байдал, ложистик гэсэн хэлтсийн scope-г тодорхойлно. Үйлдвэрийн module бүр өөр entity-тэй боловч FK relation-оор User болон Department-тэй холбогдож audit, permission, dashboard query хийх боломж бүрдүүлнэ.

**Юуг тайлбарлах вэ:**

- `User`: email, passwordHash, role, departmentId.
- `Department`: хэлтсийн нэр, өнгө, access scope.
- `Material`: агуулахын материал, stock, threshold.
- `MaterialTransaction`: материалын орлого, зарлага, хэн бүртгэсэн.
- `ProductionLog`: үйлдвэрлэлийн өдөр тутмын бүртгэл.
- `SafetyIncident`: осол, severity, status, owner.
- `EquipmentTelemetry`: rpm, load, healthScore.
- `TransportRecord`: route, delivery status.
- `AiAgentAuditLog`: AI tool болон чухал action-уудын мөр.

**Prisma ORM-ийг тайлбарлахдаа:**

> Prisma ORM нь TypeScript code болон MySQL database-ийн хооронд ажилладаг layer. SQL query-г гараар string хэлбэрээр бичихийн оронд schema, model, relation дээр үндэслэн type-safe query бичих боломж өгдөг. Мөн migration ашиглан database structure-ийн өөрчлөлтийг version control-той удирдана.

**Онцлох өгүүлбэр:**

> ERD-ийн гол санаа нь бүх module тусдаа биш, User болон Department relation-оор нэг системийн өгөгдлийн model болж холбогдсон явдал.

**Асуувал богино хариулт:**

Асуулт: "Яагаад relational database сонгосон бэ?"  
Хариулт: "Энэ системд хэрэглэгч, хэлтэс, бүртгэл, audit, permission зэрэг олон relation байгаа тул MySQL шиг relational database тохиромжтой."

### Хуудас 8. Хамгаалагдсан API ба бизнес логик

**Слайдын зорилго:**  
Request орж ирэхэд backend дээр ямар дарааллаар шалгалт хийгддэгийг тайлбарлах.

**Юу гэж хэлэх вэ:**

> Энэ слайд кодын хэрэгжилтийн гол урсгалыг харуулж байна. API request шууд database руу очихгүй. Эхлээд JWT session шалгана, дараа нь RBAC role шалгана, department permission шалгана, дараа нь Zod validation хийж input зөв эсэхийг баталгаажуулна. Үүний дараа л Prisma operation ажиллана.

**Юуг тайлбарлах вэ:**

- `JWT authentication`: хэрэглэгч нэвтэрсэн эсэх.
- `RBAC`: ямар role ямар action хийх эрхтэй вэ.
- `Department guard`: хэрэглэгч зөв хэлтсийн өгөгдөлд хандаж байна уу.
- `Zod validation`: request body, query, enum утга зөв үү.
- `Prisma operation`: findMany, create, update, delete.
- `Audit/response`: чухал action бүртгэгдээд UI руу response буцна.

**Zod validation-ийг тайлбарлахдаа:**

> Zod validation нь API руу ирж байгаа input-ийг schema-ийн дагуу шалгадаг. Жишээ нь quantity тоо байх ёстой, role зөв enum байх ёстой, required field хоосон байж болохгүй гэх мэт. Ингэснээр буруу өгөгдөл database руу орохоос өмнө таслагдана.

**Онцлох өгүүлбэр:**

> Миний API route-ууд database operation хийхээс өмнө authentication, authorization, validation гэсэн гурван хамгаалалтын давхаргаар дамжина.

### Хуудас 9. Тестлэлийн хамрах хүрээ

**Слайдын зорилго:**  
Системийн ажиллагааг ямар төрлийн тестээр шалгасныг харуулах.

**Юу гэж хэлэх вэ:**

> Кодын хэрэгжилтийг зөвхөн browser дээр хараад шалгаагүй. API, login, role permission, CRUD, dashboard refresh, protected route гэсэн гол урсгалуудыг тусад нь шалгасан. Энэ нь demo ажиллаж байгаагаас гадна backend logic зөв ажиллаж байгааг батлах зорилготой.

**Юуг тайлбарлах вэ:**

- `Postman API test`: endpoint response 200, 201, 400, 401, 403 зөв буцаж байна уу.
- `Login/auth test`: зөв user орж, буруу user татгалзагдаж байна уу.
- `RBAC role test`: ADMIN, MODERATOR, USER эрхийн ялгаа зөв үү.
- `CRUD test`: create, read, update, delete database болон UI дээр зөв шинэчлэгдэж байна уу.
- `Dashboard refresh test`: өгөгдөл шинэчлэгдэхэд dashboard refresh болж байна уу.
- `Protected API route test`: login хийгээгүй request 401/403 буцааж байна уу.

**Онцлох өгүүлбэр:**

> Тестлэлийн зорилго нь UI харагдаж байгааг биш, authentication, permission, validation, database operation зөв ажиллаж байгааг батлах юм.

### Хуудас 10. Бодит цагийн KPI ба RPM telemetry

**Слайдын зорилго:**  
Dashboard-ийн хамгийн хэрэглээний хэсгийг тайлбарлах.

**Юу гэж хэлэх вэ:**

> Энэ хэсэг нь системийн хэрэглэгчид шууд харагдах гол dashboard юм. KPI card, chart, table, alert ашиглан үйлдвэрлэл, агуулах, аюулгүй байдал, ложистикийн төлөвийг харуулна. Мөн тоног төхөөрөмжийн RPM telemetry-г health score-той холбож, ачаалал болон эрсдэлийг өнгөөр ялган харуулна.

**Юуг тайлбарлах вэ:**

- `KPI card`: гол тоонуудыг хурдан уншуулах.
- `Chart`: production trend, material movement.
- `Table`: бүртгэлийн нарийвчилсан мэдээлэл.
- `Alert`: threshold давсан үед өнгөөр ялгах.
- `RPM telemetry`: rpm, load, healthScore.
- `SWR refresh`: өгөгдлийг дахин татаж dashboard-г шинэчилнэ.
- `Recharts`: chart visualization хийхэд ашиглаж болно.

**SWR-ийг тайлбарлахдаа:**

> SWR нь frontend дээр API data fetch хийх, cache хадгалах, дахин шинэчлэхэд ашиглагддаг React data fetching арга. Dashboard дээр өгөгдөл өөрчлөгдөхөд дахин ачаалахгүйгээр шинэ төлөв авахад тохиромжтой.

**Recharts-ийг тайлбарлахдаа:**

> Recharts нь React дээр chart зурахад ашигладаг library. Production trend, material movement, KPI график зэрэг өгөгдлийг визуал хэлбэрээр харуулахад ашиглана.

**Онцлох өгүүлбэр:**

> Dashboard-ийн гол UX зорилго нь захирал олон хүснэгт уншихгүйгээр хамгийн чухал төлөвийг эхний дэлгэцээс ойлгох явдал.

### Хуудас 11. Байршуулалтын архитектур

**Слайдын зорилго:**  
Системийг local project биш, deploy хийж ажиллуулах бүтэцтэй гэдгийг харуулах.

**Юу гэж хэлэх вэ:**

> Энэ төсөл local computer дээр ажиллахаас гадна production deployment-ийн бүтэцтэй. Code GitHub repository-д хадгалагдаж, Vercel дээр deploy хийгдэнэ. Database нь Aiven Cloud дээр MySQL хэлбэрээр байрлана. Vercel environment variables ашиглан database URL, JWT secret, API key зэрэг нууц утгуудыг source code-д хадгалахгүй.

**Юуг тайлбарлах вэ:**

- `GitHub`: source code version control.
- `Vercel`: Next.js app hosting.
- `Aiven MySQL`: managed cloud database.
- `Environment variables`: secret утгуудыг code-с тусгаарлах.
- `Prisma migrate deploy`: production database schema-г update хийх.
- `HTTPS`: secure connection.

**Онцлох өгүүлбэр:**

> Нууц мэдээллийг GitHub дээр commit хийхгүй, Vercel environment variables дээр тохируулсан нь security-ийн үндсэн шаардлага.

### Хуудас 12. Эрсдэлийн менежмент

**Слайдын зорилго:**  
Системийн боломжит эрсдэлүүдийг тодорхойлж, бодит арга хэмжээ авсныг харуулах.

**Юу гэж хэлэх вэ:**

> Веб системд зөвхөн feature хийхээс гадна эрсдэлийг тооцох шаардлагатай. Энэ төсөл дээр authentication bypass, буруу input, database холболт, RPM alert хоцрох, хэрэглэгчийн буруу хэрэглээ, AI action эрсдэл зэрэг асуудлыг авч үзсэн. Эрсдэл бүрт probability, impact, mitigation буюу бууруулах арга хэмжээ тодорхойлсон.

**Юуг тайлбарлах вэ:**

- Security risk: JWT, RBAC, protected route.
- Input risk: Zod validation.
- Database risk: Prisma, migration, backup, env.
- Operational risk: alert, dashboard refresh.
- User adoption risk: энгийн UI, өнгөний ялгарал.
- AI action risk: audit log, permission guard.

**OWASP-ийг тайлбарлахдаа:**

> OWASP нь веб application security-ийн түгээмэл эрсдэлүүдийг ангилдаг стандартчилсан мэдлэгийн сан. Миний төсөл дээр injection, broken access control, sensitive data exposure зэрэг эрсдэлээс сэргийлэхийн тулд Prisma, Zod, RBAC, env variable, protected route ашигласан.

**Онцлох өгүүлбэр:**

> Эрсдэлийн менежмент нь зөвхөн жагсаалт биш, системийн архитектур болон кодын шийдэлтэй шууд холбогдсон.

### Хуудас 13. Дүгнэлт ба цаашдын зорилт

**Юу гэж хэлэх вэ:**

> Төслийн үр дүнд үйлдвэрлэлийн KPI, агуулах, аюулгүй байдал, ложистик, RPM telemetry, authentication, role-based access control, protected API, database relation, deployment зэрэг үндсэн хэсгүүдийг нэг систем болгож нэгтгэсэн. Цаашид mobile PWA, илүү нарийн analytics, forecast, notification, role permission matrix-ийг өргөжүүлэх боломжтой.

**Юуг тайлбарлах вэ:**

  - Dashboard

  - API protection

**Онцлох өгүүлбэр:**

### Хуудас 14. Үнэлгээний хүснэгт

**Слайдын зорилго:**  
Presentation нь төгсөлтийн төслийн rubric-ийн 100 онооны шалгууртай нийцэж байгааг харуулах.

**Юу гэж хэлэх вэ:**

> Сүүлийн слайд дээр үнэлгээний хүснэгтийг тусгасан. Танилцуулга 15 оноо, асуудал ба шийдэл 25 оноо, техникийн дэд бүтэц 20 оноо, кодын хэрэгжилт 25 оноо, эрсдэлийн менежмент 15 оноотой. Миний presentation энэ дарааллын дагуу зохион байгуулагдсан тул үнэлгээний шалгуур бүрийг шууд нотлох бүтэцтэй.

**Юуг тайлбарлах вэ:**

- `Танилцуулга`: зорилго, бүтэц, scope тодорхой.
- `Асуудал ба шийдэл`: тархсан өгөгдөл, dashboard solution, comparison table.
- `Техникийн дэд бүтэц`: architecture, ERD, Prisma, MySQL, Vercel.
- `Кодын хэрэгжилт`: API, JWT, RBAC, Zod, CRUD, testing.
- `Эрсдэлийн менежмент`: risk matrix, mitigation, OWASP.

**Онцлох өгүүлбэр:**

> Миний presentation-ийн бүтэц нь үнэлгээний шалгууртай шууд таарч байгаа тул аль шалгуурыг хаана хангаснаа тодорхой харуулж чадна.

## 5. Техникийн нэр томъёог асуувал ингэж тайлбарла

### Next.js App Router

Next.js App Router нь React дээр суурилсан веб application-д page, layout, server component, API route зэргийг `app` folder бүтэцтэй зохион байгуулах арга. Энэ төсөл дээр frontend page болон backend API route-уудыг нэг framework дотор ажиллуулахад ашигласан.

### TypeScript

TypeScript нь JavaScript дээр type system нэмсэн хэл. API response, database model, form input зэрэгт type ашигласнаар кодын алдааг эрт илрүүлж, project томрох үед maintain хийхэд тусалдаг.

### Prisma ORM

Prisma ORM нь TypeScript code болон MySQL database-ийн хооронд ажилладаг database layer. SQL query-г гараар string хэлбэрээр бичихийн оронд model, relation, migration ашиглан type-safe query бичих боломж өгдөг.

### MySQL / Aiven Cloud

MySQL нь relational database. Aiven Cloud нь managed database service бөгөөд database-г cloud дээр байршуулж, connection, availability, backup зэрэг operational хэсгийг хялбарчилна.

### JWT authentication

JWT буюу JSON Web Token нь хэрэглэгч login хийсний дараа session identity-г баталгаажуулахад ашиглагддаг token. API request бүр дээр хэрэглэгч нэвтэрсэн эсэхийг шалгахад хэрэглэнэ.

### RBAC

RBAC буюу Role-Based Access Control нь хэрэглэгчийн role дээр үндэслэн эрх шалгах арга. Энэ төсөлд `ADMIN`, `MODERATOR`, `USER` гэсэн 3 эрхийн түвшинтэй.

### Zod validation

Zod нь TypeScript-тэй сайн ажилладаг validation library. API руу ирж байгаа input зөв төрөлтэй, required field бүрэн, enum утга зөв эсэхийг database operation хийхээс өмнө шалгана.

### SWR

SWR нь React data fetching арга. API-с өгөгдөл татах, cache хийх, дахин шинэчлэх, dashboard refresh хийхэд ашиглана.

### Recharts

Recharts нь React дээр chart байгуулах library. KPI trend, production chart, material movement зэрэг өгөгдлийг visual хэлбэрээр харуулахад тохиромжтой.

### Modular monolith

Modular monolith гэдэг нь систем нэг codebase, нэг deploy pipeline-тэй боловч дотроо module-уудаар зохион байгуулагдсан архитектур. Энэ төсөлд frontend, API, business logic нэг Next.js project дотор байгаа.

### OWASP

OWASP нь веб application security-ийн түгээмэл эрсдэлүүдийг тодорхойлдог олон улсын мэдлэгийн сан. Энэ төсөлд broken access control, injection, sensitive data exposure зэрэг эрсдэлийг RBAC, Prisma, Zod, env variable, protected route-оор бууруулсан.

## 6. Багш нарын магадлалтай асуулт ба богино хариулт

### Асуулт: "Энэ системийн хамгийн гол үнэ цэнэ юу вэ?"

Хариулт:

> Гол үнэ цэнэ нь үйлдвэрийн олон хэлтсийн өгөгдлийг нэг dashboard дээр төвлөрүүлж, удирдлага хурдан шийдвэр гаргах боломжтой болгох явдал.

### Асуулт: "Яагаад Next.js сонгосон бэ?"

Хариулт:

> Next.js App Router нь frontend болон API route-уудыг нэг TypeScript codebase-д зохион байгуулах боломж өгсөн. Мөн Vercel дээр deploy хийхэд тохиромжтой.

### Асуулт: "Database relation хамгийн чухал хэсэг нь юу вэ?"

Хариулт:

> User болон Department relation хамгийн чухал. Учир нь role, department scope, audit, dashboard query бүгд энэ холбоос дээр тулгуурлаж ажиллана.

### Асуулт: "Security дээр юу хийсэн бэ?"

Хариулт:

> JWT authentication, RBAC role guard, protected API route, Zod validation, Prisma parameterized query, environment variables, audit log ашигласан.

### Асуулт: "Тест хийсэн үү?"

Хариулт:

> Тийм. Postman API test, login/auth test, RBAC role test, CRUD test, dashboard refresh test, protected API route test гэсэн үндсэн урсгалуудыг шалгасан.

### Асуулт: "AI эсвэл audit log яагаад хэрэгтэй вэ?"

Хариулт:

> AI action эсвэл чухал системийн үйлдэл бүрийг мөрдөхгүй бол хэн ямар өгөгдөл өөрчилсөн нь тодорхойгүй болно. Audit log нь traceability болон accountability-д хэрэгтэй.

### Асуулт: "Цааш хөгжүүлэх хамгийн эхний зүйл юу вэ?"

Хариулт:

> Production орчинд илүү бодит telemetry integration, notification system, role permission matrix, mobile PWA, analytics forecast нэмэх нь хамгийн бодит дараагийн алхам.

## 7. Хамгаалалтын төгсгөлд хэлэх дүгнэлт

Доорхыг хамгийн сүүлд хэлж болно:

> Дүгнэж хэлбэл, миний төсөл уул уурхайн тэсрэх бодисын үйлдвэрийн KPI мэдээллийг нэг dashboard дээр төвлөрүүлэх зорилготой fullstack веб систем юм. Системд Next.js, TypeScript, Prisma ORM, MySQL/Aiven, JWT authentication, RBAC, Zod validation, protected API route, KPI dashboard, RPM telemetry, Vercel deployment зэрэг үндсэн хэсгүүдийг нэгтгэсэн.
